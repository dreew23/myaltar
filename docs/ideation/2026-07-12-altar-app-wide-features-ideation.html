Grounding Context

Codebase Context

ALTAR ("DOMINION Spiritual Operating System") is a single-user Next.js 16 + Supabase spiritual-formation app. Modules: DOMINION dashboard (streaks, prayer watches, weekly commitments, Dominion Score), Sunday Pulse (6-phase weekly ritual on 13-week quarters with archive/rollover), Sermon Library (with the new 8-section sermon journal), Divine Downloads (quick capture with categories including action_required and promise), Prayer + Intercession (guided mode, request tracker, list), Declarations (recitation mode with counters), Wisdom Log (testimonies, decisions), Goals (7-goal quarter framework with a "not now" list), Settings (quarters, personal years, dual calendar lens). Server components fetch, client components write directly to Supabase.

Recurring pains observed: missing-migration errors handled by silent try/catch fallbacks (a missing table renders as "no data"); schema split across scripts/ and supabase/migrations/; Goals and Prayer History absent from the sidebar; the dashboard performs a render-time weekly_goals upsert.

External Context (web research)

YouVersion streak savers + separate cumulative badges; Lectio 365 tri-daily liturgical rhythm; PrayerMate "personal CRM for prayer" rotation; Echo Prayer's answered-prayer log (its single most-quoted feature); Bible Memory SM-2 spaced repetition; Readwise forgetting-curve resurfacing of your own highlights; 12 Week Year weekly execution scorecard (85% target, week 13 = review + reset); Psalmlog / Soul Guide voice journaling + AI "spiritual seasons" pattern detection framed as reflective mirror, not oracle; the two-app-stack finding — users pair a Bible app with one specialist, which makes ALTAR's all-in-one scope its counter-position.

Unifying thesis from generation

ALTAR captures well but almost never reads back to the user. Three capture surfaces (Downloads, Sermon Journal, Wisdom Log) have zero retrieval surfaces; several schema fields (spaced_summary, became_testimony, action_required) are written but never consumed. Resurfacing and loop-closing dominate the survivor set.

1

The Remembrance Engine — spaced resurfacing of your own captures

capture-to-retention Confidence 85% Complexity Medium
Description
A daily "stones of remembrance" card on the dashboard that resurfaces one past item — a Divine Download, a sermon journal spaced_summary, a Wisdom Log line, or a captured promise — on forgetting-curve intervals (1d / 7d / 30d / 90d). One tap responds: still alive (reschedule further out), fulfilled (convert a promise to a testimony), or archive. Promises additionally get an optional maturity horizon (a quarter, a personal-year boundary) and are re-presented when they come due.
Basis
directThe sermon journal collects a field literally labeled "One-line summary for spaced review" (components/app/sermons/sermon-journal.tsx, line 425) that nothing in the app ever displays again — verified by codebase-wide search. The downloads page fetches reverse-chronological only; category: "promise", became_testimony, and testimony_note exist on the record with no mechanism that ever moves a promise toward them. externalReadwise's daily resurfacing of one's own highlights; Bible Memory's SM-2 scheduling (2M users, 30M verses).
Rationale
This inverts the capture economics of the whole app: today every capture depreciates the moment it scrolls off; with resurfacing, every capture becomes an appreciating asset — which retroactively raises the value of the Quick Capture FAB, the sermon journal's Revelation section, and the promise category. The schema already promises this feature; the UI is the only missing half.
Downsides
Needs a small scheduling table and a review-state model; a badly tuned cadence could make the card feel like nagging. Start with one card per day, never more.
2

Close the Capture Loop — action-required inbox + weekly reconciliation

daily-engagement-loop Confidence 88% Complexity Low–Medium
Description
Three connected fixes for the same dead end. (1) A dashboard card lists open action_required downloads sorted by age, with visual aging (7+ days = amber) and one-tap resolve or convert. (2) Pulse Phase 5 opens with the week's un-actioned action_required / direction downloads as one-tap priority candidates — promote, defer, or dismiss — instead of three blank text inputs. (3) An accounting-style close: Pulse Phase 6 asks that each open item be scheduled, converted, or consciously written off before the week's books close.
Basis
directAll four claims verified in code: the dashboard fetches divine_downloads as select("id") count-only; fetchWeekDownloads() in lib/pulse-page-data.ts selects only "id, content, created_at" — dropping the category field before Pulse ever sees it; priorityGoals in phase5-plan.tsx (line 40) is local state whose dropdown selections are silently lost; the Phase 6 close checklist is purely logistical (close tabs, sticky note). externalReadwise's core insight — capture without scheduled resurfacing is worthless; month-end close from accounting.
Rationale
A category literally named "action required" currently produces no action anywhere in the system. The capture pipe, the taxonomy, and the Pulse hub all exist — one join turns a notes app into an obedience tracker, and trust in capture compounds every week the loop visibly works. Cheapest high-impact item on this list.
Downsides
The Phase 6 reconciliation gate could feel heavy on busy weeks — make "write off with a reason" a fast, guilt-free path, or the gate will train avoidance of Pulse itself.
3

Kill the Backfill — evidence-derived Sunday Pulse

weekly-quarterly-rhythm Confidence 82% Complexity Medium
Description
Pulse Phase 2 currently instructs "Assess from data, not memory" — then hands you a manual 7-day backfill grid. Invert it: derive each cell from records the app already holds (prayer session logs, declaration logs, downloads captured, sermon journal touches), render the grid pre-filled, and reserve manual edits for genuine offline exceptions. Human input narrows to the one thing only a human knows: the energy score.
Basis
directVerified: components/app/pulse/phase2-measure.tsx line 86 reads "Ensure all data is logged before reviewing. Assess from data, not memory." while the component's mechanism is a manual onBackfillDay grid — and prayerSessionDates (the exact data needed to fill those cells) is already in the component's props, used only to display a count.
Rationale
The weekly audit is the hinge of the entire 13-week system; if its inputs are reconstructed memories, every downstream verdict (Phases 3–6, quarter review, and any future scorecard) inherits the distortion. The app's own on-screen motto is contradicted by its own implementation — and the fix is mostly queries over tables that already exist. This is also the prerequisite that makes idea 4's scorecard trustworthy.
Downsides
Requires agreeing on what counts as evidence for each checkbox (does one declaration tap count as "declarations done"?). Define the thresholds once, visibly, and let manual override remain for edge cases.
4

Honest Numbers — a true streak and an execution scorecard

daily-engagement-loop Confidence 85% Complexity Medium
Description
Fix the two headline metrics that are currently misleading. (1) The dashboard "prayer streak" becomes a true consecutive-day computation over full history, with YouVersion-style grace days (one auto-applied saver per week) and a separate cumulative-days count that only ever grows. (2) The Dominion Score — currently an arbitrary blend of the broken streak and sermon count — becomes a 12 Week Year execution score: percent of the week's own weekly_commitments kept, with the 85% line as the "great execution" marker, displayed daily and rendered as 13 tiles per quarter.
Basis
directVerified: the streak is devotionsRes.data?.filter((d) => d.prayer_complete).length over a query bounded .gte("date", startOfWeekStr) (app/app/dashboard/page.tsx line 172) — a non-consecutive count capped at 7 that silently resets every Monday. The Dominion Score formula (line 175) is Math.min(10, Math.round((prayerStreak/7)*5 + (sermonsThisWeek/5)*5)), ignoring the weekly commitments fetched on the same page. externalYouVersion streak savers + cumulative badges; 12 Week Year's weekly scorecard with 85% leading-indicator target.
Rationale
The two most prominent numbers on the daily command center are mathematically wrong or arbitrary. A score you can't influence through your stated commitments trains you to ignore it; a streak that punishes a logging omission (the most common failure for a solo user) is pure demotivation with zero information. Grounding both numbers in the user's own covenant makes them the honest leading indicators the 13-week system is supposed to run on.
Downsides
Persisting weekly scores needs a small table; changing the headline metric changes what the app "rewards," so decide deliberately what belongs in the score.
5

The Answered-Prayer Pipeline — from request to testimony

prayer-intercession-lifecycle Confidence 80% Complexity Medium
Description
Give prayer requests CRM-style stages (requested → praying → answered → testimonied) with last-prayed recency. Marking a request answered auto-drafts a Wisdom Log testimony pre-filled with the original request, dates asked and answered, and elapsed time — edit and confirm rather than re-type in another module. On the intake side, a computed daily queue deals the 2–3 most-neglected entries into the matching prayer watch, PrayerMate-style, so nobody silently falls off the list; praying from guided mode updates recency automatically.
Basis
externalEcho Prayer's answered-prayer log is its single most-quoted feature; PrayerMate's category-rotation scheduling is the proven "personal CRM for prayer" mechanic. directThe verifier confirmed both ends exist in-app (answered-prayer flow with answer_notes; wisdom_entries; became_testimony on downloads) with no pipe between them — and found last_prayed_at already exists on the prayer-request table, updated by prayer-mode-flow.tsx, which strengthens feasibility.
Rationale
Answered prayers are the one data type whose value strictly grows with time — each becomes evidence for future intercession. The moment of marking "answered" is the highest-motivation instant to capture a testimony, and the system already holds every field. This also unifies three modules (/app/prayers, /app/prayer, /app/journal) that currently don't feed each other — connective tissue no two-app stack can replicate.
Downsides
Stage modeling needs a small migration on the requests table; the dealt-hand queue should start as a suggestion, not a lock, to preserve spontaneity in prayer.
6

The Season Letter — a quarterly mirror in your own words

cross-module-intelligence Confidence 70% Complexity High
Description
At week 13 (and optionally mid-quarter), generate a reflective summary of the quarter built strictly from the user's own words across modules — Pulse notes, sermon Burden/Revelation sections, wisdom entries, downloads: recurring themes, the emotional arc, promises captured versus promises acted on, a proposed "spiritual season" label to confirm or edit into the quarter archive. Framed explicitly as a mirror ("here is what you wrote"), never an oracle ("here is what God is saying"), with every claim linking back to its source entry.
Basis
directVerified: fetchLastWeekTimeAnalysis() is limit(1).maybeSingle() — only the single most recent reflection is ever read again; 13 weeks of written self-analysis per quarter currently evaporates. externalPsalmlog and Soul Guide's "spiritual seasons" pattern detection is the 2025–26 frontier of the category, with "reflective mirror, not oracle" as the established design guardrail.
Rationale
This is the payoff of the all-in-one bet: no specialist app holds cross-module text to mine, so this is the one feature on this list no commercial product can offer you. Its value grows fastest with data — modest in quarter one, startling after four. It also gives quarter close an emotional climax the rollover flow currently lacks, and confirmed season labels become structured data for every future retrospective.
Downsides
Highest complexity here (an LLM pass over personal text, prompt design, a review/confirm UI). Sequence it after the cheaper loops (ideas 1–4) are generating cleaner data to mine.
7

The Schema Doctor — fail loud, name the missing migration

cross-module-intelligence Confidence 90% Complexity Low
Description
A settings-page (or startup) probe that checks the live Supabase schema against a declared manifest of expected tables and columns, and renders a "System Integrity" banner naming exactly what is missing and which file in supabase/migrations/ creates it — with copyable SQL. Replaces the app-wide pattern of silent catch-return-empty, where a missing table renders as "no data," indistinguishable from an empty life.
Basis
directAll three citations verified: app/app/dashboard/page.tsx lines 152–153 wrap ten queries in one try with catch {'{'} // Tables may not exist; dashboard still renders with zeros {'}'}; new-quarter-flow.tsx lines 90–97 regex-match the error message (/archived_at/i.test(deErr.message)) to hand-patch one column's drift; app/app/journal/page.tsx line 31 documents its own drift in a comment naming the exact migration file to run.
Rationale
Missing-migration errors are this project's single most recurring real-world pain (they have surfaced repeatedly during development), and the app is architected to hide them — turning two-minute fixes into confusing debugging sessions. Per-column regex patches cannot scale. One diagnostic surface replaces an unbounded family of future fallback hacks, and every feature above this line adds tables that would otherwise add new silent failure modes.
Downsides
The manifest must be kept in sync with migrations — generate it from the migrations folder rather than maintaining it by hand.
